# Recrova 使い方ガイド

Recrova（ReciPalMock）は、経費・勤怠・休暇申請をまとめて扱う社内ワークフロー業務のモックアプリです。実データは保存されません。デモ環境として、画面遷移・申請フロー・承認導線・権限による表示制御を体験できます。

公開デモ: <https://wacca.github.io/ReciPalMock>

---

## 1. ログイン

1. ログイン画面でユーザー ID とパスワードを入力します。
2. 「ログイン」を押すとダッシュボードに遷移します。

> モック環境のため、ユーザー ID は任意の文字列で受け付けます。パスワードは入力されていれば内容は問いません。
> 画面下部の「デモアカウント」一覧から ID をワンクリックでセットできます。各アカウントには事前に役割（ロール）が設定されています。

## 2. 画面の基本構成

- **左サイドレール（PC）／下部ナビ（モバイル）**: メニュー切り替え。ロールに応じて表示項目が変わります。
- **トップストリップ**: 画面タイトル・パンくず・ユーザー情報・ログアウト・ロール切替。
- **コマンドパレット**: `Ctrl + K` （macOS は `⌘ + K`）でどこからでも起動。画面名や「新規 経費申請を作成」などのアクションを横断検索できます。
- **ダッシュボード**: ログイン直後に表示されるホーム。自分のタスクや承認待ち、最近の申請状況を一覧できます。

### ロール切替（デモ用）

トップストリップのロールバッジから、表示確認用に一時的にロールを切り替えられます。元のアカウントの権限以下に変更しても画面メニューが動的に絞り込まれるので、ロール毎の見え方を確認できます。

## 3. ロールと権限

| ロール | できること |
| --- | --- |
| 一般社員 | 自分の申請・勤怠入力 |
| 承認者 | 一般社員 + 部下の承認業務 |

メニューおよびルーティングは権限ベースで自動的に絞り込まれます。

## 4. 機能カテゴリ別ガイド

### 4.1 自分の申請

#### 経費申請（`/application`）
- 経費の新規作成・下書き保存。
- コマンドパレットの「新規 経費申請を作成」からも直接開けます。

#### 経費履歴（`/submitted`）
- 自分が提出した経費申請の状態（下書き／申請中／承認済／差戻し）を確認できます。

#### 勤怠申請（`/leave-application`）
- 休暇・時間休・遅刻・早退などの申請を作成します。

#### 勤怠申請履歴（`/leave-submitted`）

#### 月次勤怠入力（`/attendance-input`）

承認画面では、承認・差戻しの操作と申請内容の詳細表示ができます。経費承認の明細テーブルには「領収書」列があり、サムネクリックでプレビューダイアログが開きます（[6. 領収書のアップロードとプレビュー](#6-領収書のアップロードとプレビュー) 参照）。

## 5. 申請ワークフローと編集可否（一般原則）

経費・勤怠・休暇いずれの申請も、以下のステータス遷移と編集ルールに従う想定です。本モックではすべての画面で同じ考え方を採用しています。

### 5.1 ステータス遷移

```
[下書き] ──提出──▶ [申請中] ──承認──▶ [承認済]
   ▲                  │
   │                  └─差戻し──▶ [差し戻し] ──修正・再提出──▶ [申請中]
   │
   └──取り下げ─────────┘
```

### 5.2 ステータス別の編集可否

| ステータス | 申請者本人 | 承認者 | 可能なアクション |
| --- | --- | --- | --- |
| **下書き** | ◯ 編集可 | — | 編集 / 削除 / 提出 |
| **申請中（承認待ち）** | ✕ 編集不可 | ✕ 編集不可 | 申請者は「取り下げ」のみ。承認者は「承認」「差戻し」 |
| **差し戻し** | ◯ 編集可 | — | 修正のうえ再提出、または取り下げ |
| **承認済** | ✕ 編集不可 | ✕ 編集不可 | 参照のみ（修正は取消申請など別フロー） |

### 5.3 原則

- **承認内容と実データの整合性**: 承認者がレビュー中に内容が変わると「承認した内容」と「実際のデータ」が食い違うため、申請中は編集を禁止します。
- **監査証跡の保持**: 申請時点・承認時点のスナップショットを残すため、確定後は編集不可とします。
- **状態遷移は明示的なアクション経由のみ**: 「取り下げ」「差戻し」「再申請」など、誰が何をしたかを記録できる経路でのみ状態を変更します。
- **修正が必要な場合**: 申請中のデータを直接書き換えるのではなく、「取り下げ → 編集 → 再申請」または「差戻し → 修正 → 再提出」の経路で対応します。

### 5.4 承認取消（誤承認のリカバリ）

承認操作を誤って実行してしまった場合に備え、本アプリでは **承認取消（Unapprove）** を提供しています。

- **対象画面**: 経費承認 / 勤怠申請承認 / 月次勤怠承認 の各「承認履歴」タブ
- **操作できる人**: 承認した本人 + 上位ロール（HR / 総務、システム管理者）
- **対象ステータス**: 「承認済」のみ（差戻し済・取消済は対象外）
- **必須項目**: 取消理由（コメント）
- **遷移先**: 「申請中」へ戻し、再度承認 / 差戻しを判断できる状態にします
- **制約**: 後続処理（外部 SaaS 連携・月次締め）が完了している場合は取消不可。その場合は「取消申請」を別途起票する運用とします。
- **副作用のロールバック**: 勤怠申請承認の場合、勤怠への反映分も同時に取り消されます

承認取消を実行すると、申請者と関係者に通知され、操作履歴が積み上げ式の監査ログに記録されます（5.5）。

### 5.5 操作履歴（監査証跡）

申請のライフサイクル上で発生したイベントは、**積み上げ式（イベントログ方式）** で全件保持します。最終更新者のみを保持する方式ではなく、誰が・いつ・何を・なぜ実行したかを時系列で復元できる構造です。

#### 履歴エントリの構造

| 項目 | 内容 |
| --- | --- |
| `eventType` | `submit` / `approve` / `reject` / `sendback` / `withdraw` / `unapprove` / `edit` |
| `actorId` / `actorName` | 実行者 |
| `actorRole` | 実行時点のロール（後でロールが変わっても証跡として固定） |
| `fromStatus` / `toStatus` | 遷移前後のステータス |
| `comment` | 理由・コメント（差戻し・承認取消では必須） |
| `at` | 実行日時（ISO8601） |

#### 設計方針

- **追記のみ（immutable）**: 既存エントリは UPDATE / DELETE しない。修正したい場合も「修正イベント」を新規に追記する。
- **本体側は現在ステータスのみ保持**: 申請テーブルには `status` と `updated_at` だけ持ち、履歴は別配列 `history` に積む。検索・参照は本体側、監査・タイムライン表示は履歴側、と責務を分離する。
- **画面表示**: 承認履歴カードに展開可能なタイムラインを表示し、「○○が承認を取り消しました（理由: △△）」のような単位で時系列に追える。
- **テキストログとの違い**: アプリログへの出力は補助的なデバッグ用途。正式な証跡は DB（モックでは `localStorage`）に構造化して保持する。

## 6. 領収書のアップロードとプレビュー

経費申請の各明細行には領収書を 1 件添付できます。**画像（JPEG / PNG / HEIC / WebP / GIF / SVG）と PDF の両方** を受け付けます。スマホで撮影した写真も、ECサイトやメール添付で受け取った PDF もそのまま添付できる想定です。

### 6.1 申請者側（経費申請画面）

1. 明細行のアップロードボタン「**領収書をアップロード（画像 / PDF）**」を押下。
2. `accept="image/*,application/pdf"` で対象拡張子のみ選択可能。対象外を選んだ場合は警告 Snackbar を表示し、入力をクリアします。
3. アップロード成功後はサムネ（56px）と削除アイコンが表示されます。
   - 画像: 実際のイメージを縮小表示。
   - PDF: 赤系の PDF アイコン。
4. サムネクリックで原寸プレビューダイアログが開きます。

### 6.2 承認者側（経費承認画面）

#### 承認待ちタブ

申請単位のカード内、明細テーブルに「**領収書**」列が並びます。

- **サムネ常時表示**（36px）。画像は実イメージ、PDF は PDF アイコン。
- 添付なしの行は「—」プレースホルダ。
- サムネクリックで `ReceiptPreviewDialog` が開きます。
- プレビュー中は承認・差戻しのショートカット（`A` / `R`）が無効化され、誤操作を防ぎます。

#### 承認履歴タブ

過去に承認・差戻しした申請の判断根拠を後から再確認するため、各カードに **「明細 (N) 領収書 M」** トグルボタンを追加しています。

- クリックで明細テーブルが展開（「操作履歴」とは独立に開閉可）。
- 明細テーブルには承認待ちタブと同じ領収書列を表示。
- 監査や承認取消の判断時に、当時アップロードされていた領収書をそのまま確認できます。

### 6.3 経費履歴（申請者）と経費申請検索（管理者）

| 画面 | 領収書の見え方 |
| --- | --- |
| 経費履歴 (`/submitted`) | カード展開時の明細テーブルに領収書列を表示。サムネクリックでプレビュー（カードのトグルとは独立して動作） |
| 経費申請検索 (`/expense-search`) | 明細単位の検索結果テーブルに領収書列（32px サムネ）を表示。CSV 出力にも「領収書ファイル」列を追加 |

申請者は過去の自分の領収書をいつでも見返せ、HR / 総務は横断検索結果から直接領収書を確認できます。CSV にはファイル名のみ含めるため、画像/PDF 本体は連携 SaaS や別途オブジェクトストレージ側で管理する想定です。

### 6.4 プレビューダイアログの共通仕様

| 機能 | 画像 | PDF |
| --- | --- | --- |
| インライン表示 | `<img>` を中央配置 | `<iframe>` でブラウザ標準ビューア |
| 90°回転 | ◯ | — |
| 回転リセット | ◯ | — |
| 新しいタブで開く | ◯ | ◯ |
| ダウンロード | ◯ | ◯ |
| 閉じる | ◯ | ◯ |

種別の判定は `receiptMimeType`（`application/pdf` か `image/*`）で行います。MIMEが取れないレガシーデータはファイル名の拡張子から推測します（`expenseApplicationStore.js` の `inferMimeTypeFromName`）。

### 6.5 サンプルデータ

`public/sample-receipts/` 配下にデモ用の領収書を配置しています。

| ファイル | 種別 | 用途 |
| --- | --- | --- |
| `sample-taxi.svg` | 画像 (SVG) | タクシー領収書 |
| `sample-meal.svg` | 画像 (SVG) | 飲食店レシート |
| `sample-train.svg` | 画像 (SVG) | JR 領収書 |
| `sample-receipt.pdf` | PDF | 書籍購入の領収書（最小構成 PDF） |

サンプル PDF は `scripts/gen-sample-pdf.py` で再生成できます（Python 3 必要）。

### 6.6 データ構造

明細行（`details[i]`）に以下のフィールドを保持します。

| フィールド | 内容 |
| --- | --- |
| `receiptName` | 元ファイル名（拡張子つき） |
| `receiptPreview` | プレビュー用 URL。アップロード時は `URL.createObjectURL` の `blob:` URL、サンプルデータは相対 URL |
| `receiptMimeType` | `image/*` または `application/pdf`。空の場合は `receiptName` から推測 |

> モック実装のため、`blob:` URL はページリロード後は復元されません。永続化が必要な業務実装では、別途オブジェクトストレージ（S3 など）への保存と署名付き URL の払い出しを行う想定です。詳細な保存・監査要件は連携 SaaS 側に委ねます。

## 7. 便利な操作

- **コマンドパレット (`Ctrl/⌘ + K`)**: 画面名で直接ジャンプ。下書き作成などのアクションも実行可能。直近のジャンプ履歴は記憶されます。
- **モバイル表示**: 画面サイズに応じてサイドレールがハンバーガー＋下部ナビに切り替わります。
- **ログアウト**: トップストリップ右上のアカウントメニューから。

---

## 8. 注意事項

- 本アプリはモック環境です。入力したデータは画面を閉じる／リロードすると失われます（一部の UI 状態は `localStorage` に保存されます）。
- 認証・承認・SaaS 連携などはすべてダミー動作です。本番業務には使用しないでください。
